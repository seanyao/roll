## 一、核心思路：反馈驱动的持续交付

CNX 的核心很简单：**设定目标 → 执行 → 检查结果 → 根据反馈调整**，持续循环。

```
┌──────────────────────────────────────────────────────────────┐
│                    反馈驱动循环                                │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   目标              感知               比较                   │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│   │   BACKLOG   │←─│  Sentinel   │←─│   Diff      │         │
│   │  (期望状态)  │  │  (实际状态)  │  │  (差距分析)  │         │
│   └──────┬──────┘  └─────────────┘  └──────┬──────┘         │
│          │                                  │                │
│          ▼                                  ▼                │
│   ┌─────────────┐                    ┌─────────────┐         │
│   │   Design    │                    │    Fix      │         │
│   │   设计+规划  │                    │   修复改进   │         │
│   └─────────────┘                    └─────────────┘         │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

**工作流阶段:**

```
Design (讨论+规划) → Build (执行) → Check (巡检) → Fix (修复) → 循环
    $cnx-design       $cnx-story-build   $cnx-sentinel     $cnx-fix-build
```

## 三、未来方向：多智能体协同与 bridge.ai

> 🔮 **理论已设计，落地载体：bridge.ai**

### 3.1 落地载体：bridge.ai

**bridge.ai** 是这套理论的**多智能体协同平台**。

```
bridge.ai 架构：
═══════════════════════════════════════════════════════════

                    ┌─────────────┐
                    │   Human     │  ← 设定目标、做决策
                    │  (人类)     │
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
              │     bridge.ai          │  ← 多智能体协同平台
              │   (协同编排层)          │
              │  ┌──────────────────┐  │
              │  │  Agent Router    │  │  ← 任务分发
              │  │  (智能路由)       │  │
              │  └────────┬─────────┘  │
              │           │            │
              │  ┌────────┼────────┐   │
              │  ▼        ▼        ▼   │
              │ ┌────┐  ┌────┐  ┌────┐ │
              │ │Claude│  │Codex│  │Kimi│ │  ← 多智能体执行
              │ │规划 │  │编码 │  │执行│ │
              │ └──┬─┘  └──┬─┘  └──┬─┘ │
              │    │       │       │   │
              │    └───────┴───────┘   │
              │           │            │
              │    ┌──────┴──────┐     │
              │    │  Shared     │     │  ← 共享上下文
              │    │  Context    │     │     (BACKLOG.md)
              │    │  (Memory)   │     │     (AGENTS.md)
              │    └─────────────┘     │
              └────────────────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │   Project   │  ← 实际项目
                    │   (GitHub)  │
                    └─────────────┘
```

### 3.2 bridge.ai 的核心能力

| 能力 | 说明 | 对应理论 |
|------|------|---------|
| **Agent Router** | 根据任务选择最适合的智能体 | 智能调度 |
| **Shared Context** | 多 Agent 共享的上下文和记忆 | 统一目标 |
| **Handoff Protocol** | Agent 间标准化的交接协议 | 多代理编排 |
| **Feedback Loop** | 执行结果的反馈和验证 | 反馈驱动 |
| **Cost Control** | 多 Agent 调用的成本优化 | 工程化实践 |

### 3.3 愿景：超级智能体协同

```
未来愿景（通过 bridge.ai 实现）：
═══════════════════════════════════════════════════════════

                    ┌─────────────┐
                    │   Human     │  ← 设定目标
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
              │     bridge.ai          │  ← 多智能体协调平台
              │   (Agent Router)       │
              │                        │
              │  ┌──────────────────┐  │
              │  │  任务分析        │  │  ← 理解需求
              │  │  → 选择 Agent    │  │  ← 匹配最佳智能体
              │  │  → 分发执行      │  │  ← 协同工作
              │  │  → 整合结果      │  │  ← 统一输出
              │  └──────────────────┘  │
              │                        │
              │  ┌──────────────────┐  │
              │  │  可选智能体       │  │  ← 支持任意 Agent
              │  │  (Agent Options) │  │     不预设分工
              │  │                  │  │
              │  │  • Kimi Code CLI │  │     本地执行
              │  │  • Claude        │  │     架构推理
              │  │  • Codex         │  │     IDE集成
              │  │  • Gemini        │  │     多模态
              │  │  • 其他...       │  │     持续扩展
              │  └──────────────────┘  │
              └────────────────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │   项目      │  ← 实际执行
                    └─────────────┘
```

**关键原则**：
- bridge.ai **不预设** Agent 分工（如"Claude 只能做架构"）
- bridge.ai **根据任务**动态选择最适合的 Agent
- 所有 Agent 都是平等的 **Option**，通过能力注册接入
- Human 可以指定 Agent，也可以让 bridge.ai 自动选择

### 3.2 理论设计：多 Agent 分工

| 智能体 | 理论角色 | 理论能力 | 当前状态 |
|--------|---------|---------|---------|
| **Claude** | 架构师/规划师 | 系统设计、复杂推理 | 🔮 未接入 |
| **Codex** | 实现者 | 代码生成、IDE集成 | 🔮 未接入 |
| **Kimi** | 执行者 | 当前使用 | ✅ 已实现 |

### 3.3 多 Agent 编排模式（理论）

**模式一：Pipeline（串联）**
```
Claude (设计) → Codex (编码) → Kimi (执行)
```

**模式二：Expert Panel（会诊）**
```
         ┌── Claude (架构)
问题 →  ├── Codex (实现)  → 决策
         └── Kimi (执行)
```

**模式三：Master-Worker（主从）**
```
Claude (Master)
    ├── 委派 Codex: "实现API"
    └── 委派 Kimi: "运行测试"
```

3. **无 Agent 间通信**：Claude/Codex/Kimi 未联动
4. **能力边界**：复杂架构设计仍依赖人类
